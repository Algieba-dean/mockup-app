## Anti-Patterns Verdict

### LLM Assessment

The interface does NOT look AI-generated. The editorial monochrome approach is genuinely distinctive and well-executed — the constraint of zero color creates a strong brand identity that avoids SaaS clichés. The Playfair Display + Geist pairing on the canvas is effective. The overall layout (sidebar-canvas-sidebar) follows professional tool conventions (Figma, Linear).

However, the interface suffers from **functional thinness** — it looks premium but the interaction layer is skeletal. No keyboard shortcuts, no undo, no contextual help, no progressive disclosure in the sidebar. This is the opposite of AI slop: it's a beautiful shell without the interaction depth that a pro tool demands.

### Deterministic Scan

1 finding detected:

- **`layout-transition`** (`src/index.css:277`): `.sidebar` animates `width` with CSS `transition`. This causes layout thrash on sidebar collapse/expand. Should use `transform: translateX()` or `grid-template-columns` transition instead.

No gradient text, side-stripe borders, glassmorphism-as-default, or other absolute bans detected. The codebase is clean from an anti-pattern perspective.

## Overall Impression

MockupApp has a **striking visual identity** — the editorial monochrome system is confident and distinctive. The canvas rendering with Fabric.js is technically sound. But the **interaction design is at prototype level**: no undo, no keyboard shortcuts, no progressive disclosure, no empty state guidance, and the right sidebar dumps 12+ control groups into one scrollable column without any hierarchy or collapsibility. For a tool whose PRODUCT.md promises "Frictionless Velocity" and "Contextual Intelligence", the current UX is closer to a static property inspector than a professional creative tool.

The single biggest opportunity: **progressive disclosure and interaction depth**. The UI looks like Linear but behaves like a settings panel.

## What's Working

1. **Monochrome design system is well-executed.** The `index.css` token system is clean, the dark/light theme toggle works via CSS variables, and the 1px border separations create a genuinely editorial feel. The constraint is purposeful, not lazy.

2. **Canvas rendering architecture is solid.** The `canvasManager.ts` handles multi-device layouts, dynamic island rendering, frosted glass panels, and responsive scaling via the `R` ratio factor. The off-screen canvas cropping technique for screenshots is professional-grade.

3. **Multi-page export workflow is complete.** The ZIP export with multiple device presets (iOS 6.9", 5.5", iPad, Android) is a genuinely useful feature that serves the core product promise. The export modal with checkbox selection is clear and functional.

## Priority Issues

### [P1] Right sidebar cognitive overload — 12+ control sections in a single scroll

**What**: The `RightPropertiesPanel` renders 画布背景, 视觉特效, 设备布局预设, 多机型混搭与三维偏置, and 文案与排版 as one continuous scroll. That's 15+ distinct sliders, 4 select dropdowns, 8+ buttons, and 6 text inputs — all visible at once.

**Why it matters**: Violates the Working Memory Rule (≤4 items per group). Users doing one task (e.g., adjusting device angle) see controls for unrelated tasks (e.g., background gradient, subtitle font). The PRODUCT.md principle "Frictionless Velocity" is undermined when every property is always visible.

**Fix**: Group into collapsible accordion sections. Default-open only the section relevant to the current selection context (e.g., if a device is selected, open the device section). Each section header shows a compact preview of current values.

**Suggested command**: `/impeccable layout src/components/RightPropertiesPanel.tsx`

### [P1] No undo/redo system

**What**: There is zero undo/redo capability. Every slider change, text edit, device transform, and page deletion is permanent until browser refresh (which also loses everything).

**Why it matters**: Core product principle is "Absolute Precision" — but without undo, one wrong slider drag can ruin a carefully set up composition with no way back. This is the #1 reason a power user would distrust the tool.

**Fix**: Implement a state history stack (last 20-30 states) with Ctrl+Z / Ctrl+Shift+Z. The `pages` state array is the single source of truth now, making this architecturally straightforward.

**Suggested command**: `/impeccable harden src/App.tsx`

### [P1] Zero onboarding or empty state guidance

**What**: On first load, the user sees a canvas with an empty device frame, two sidebars full of controls, and no indication of what to do first. The left sidebar shows "暂未导入截图素材" but doesn't guide the user toward the next action. The bottom dock shows "空白画幅" without explaining the concept.

**Why it matters**: PRODUCT.md says "Frictionless Velocity: Actions precede options." But a new user has no idea that their first action should be dragging a screenshot onto the canvas. The tool does not teach by doing.

**Fix**: Add an empty-state overlay on the canvas viewport that says "拖拽应用截图到此处开始" with a dashed upload area. After the first screenshot is uploaded, show a brief inline hint about the sidebar controls. Keep it minimal — one sentence, dismissible.

**Suggested command**: `/impeccable onboard src/components/CanvasViewport.tsx`

### [P2] Sidebar `width` animation causes layout thrash

**What**: `src/index.css:277` animates `transition: width` on sidebar collapse. This triggers layout recalculation on every animation frame.

**Why it matters**: Jank during sidebar toggle, especially noticeable when the Fabric.js canvas is rendering complex scenes. The detector flagged this.

**Fix**: Replace `width` transition with `grid-template-columns` transition on `.app-main`, or use `transform: translateX(-100%)` to visually hide sidebars without layout reflow.

**Suggested command**: `/impeccable optimize src/index.css`

### [P2] No state persistence — all work lost on page refresh

**What**: The `pages` array, uploaded screenshots, and all configuration live in React state only. A page refresh (or accidental browser close) wipes everything.

**Why it matters**: PRODUCT.md explicitly promises "Zero Lock-In & Total Privacy: Projects are saved locally (via IndexedDB) and run offline-first." This is not implemented. A user who spends 10 minutes setting up a 5-page mockup sequence will lose everything on an accidental F5.

**Fix**: Serialize `pages` + `screenshots` to IndexedDB (or localStorage for MVP) on every change. Hydrate on app load. Add a visual "saved" indicator.

**Suggested command**: `/impeccable harden src/App.tsx`

## Minor Observations

- **`z-index` values are arbitrary**: `9998`, `9999`, `15`, `20` used in App.tsx and CanvasViewport without a semantic scale. Should be tokenized (`--z-overlay`, `--z-modal`, `--z-toast`).
- **Google Fonts loaded via CSS `@import`**: `index.css:1` loads 6 font families synchronously. This blocks first paint. Should use `<link rel="preload">` or `font-display: swap` at minimum.
- **No `prefers-reduced-motion` media query**: The CSS has transitions everywhere but no reduced-motion alternative, violating WCAG 2.1 SC 2.3.3.
- **Canvas hardcoded to 1242×2208**: Not a bug per se, but limits future flexibility. The export already supports multiple sizes; the preview canvas should match the selected export target.
- **`confirm()` / `alert()` / `prompt()` used for dialogs**: Three browser-native dialogs break the premium aesthetic. Should be custom inline modals using the existing `ds-panel` component.
