## Priority Issues

### [P2] Tooltips have no discovery affordance

**What**: The 7 new tooltips (`RightPropertiesPanel.tsx` lines ~764-939) are native `title` attributes on plain text labels. There's no visual cue — no "?" icon, no dotted underline, nothing — telling a user that hovering will reveal help.

**Why it matters**: Heuristic #10 credit is capped because "help exists" isn't the same as "help is found." Jordan (first-timer) has no reason to hover over "三维错切" rather than just guessing or skipping it.

**Fix**: Add a small `?`/info icon next to the label (or a dotted-underline text-decoration) so the affordance is visible before the hover, not just during it.

**Suggested command**: `/impeccable clarify src/components/RightPropertiesPanel.tsx`

### [P2] Tooltip coverage is inconsistent within the same section

**What**: "设备布局与变换" mixes tooltip-covered sliders (三维错切, 截图垂直偏移, etc.) with others that still have none, in the same accordion.

**Why it matters**: Inconsistent help coverage inside one visual group reads as arbitrary rather than deliberate — users can't predict which controls will explain themselves.

**Fix**: Audit the remaining sliders in that section and extend the same tooltip pattern, or explicitly decide which ones are self-explanatory enough to skip.

**Suggested command**: `/impeccable clarify src/components/RightPropertiesPanel.tsx`

### [P3] No power-user accelerators beyond undo (carried over, unchanged)

**What**: Zoom is still 28×28px button-click only, page reordering in `AssetDock.tsx` has no drag-and-drop, and every file input is single-file (no `multiple`).

**Why it matters**: Alex (power user) hits a ceiling within minutes of real use; this was flagged in both prior runs and hasn't regressed but also hasn't moved.

**Fix**: Scroll-wheel/pinch zoom, `draggable` reordering in the page dock, `multiple` on the screenshot file input for batch-adding pages.

**Suggested command**: `/impeccable optimize src/components/CanvasViewport.tsx` (zoom) + `/impeccable craft` for page reordering

### [P3] Sidebar collapse still transitions `grid-template-columns` (evaluated, not changed)

**What**: `.app-main` (`index.css:287`) animates `grid-template-columns` on sidebar collapse — the same item flagged last run.

**Why it matters**: Investigated this run: a pure `transform`-based fix (as generically suggested) can't reclaim layout space for sibling content — that's inherent to compositor-only transforms, not an implementation gap. Reproducing the current "canvas expands into freed space" behavior requires a layout-affecting transition; this is the same trade-off VS Code/Figma make for collapsible panels.

**Fix**: Leave as-is unless profiling shows real jank (infrequent user-triggered toggle, not a continuous animation). Not worth the regression risk of a transform-only rewrite for this specific behavior.

**Suggested command**: none — accepted trade-off, downgraded from an actionable item.
